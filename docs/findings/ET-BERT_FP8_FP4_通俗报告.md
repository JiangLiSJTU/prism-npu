# 低精度（FP8/FP4）值不值？——昇腾 910B 真机实验回答四个架构问题

> 读者为技术读者，但不熟悉本项目的内部工具与历史数据，故方法与口径都在文内交代清楚。
> 实验载体：ET-BERT（BERT-base 拓扑，vocab=256，L=12，d=768，S=128，约 85M 参数）。
> 实验平台：昇腾 910B4。日期：2026-06-05。结论全部基于真机实测。

---

## 摘要

我们在 910B4 上实测了 ET-BERT 在 **FP16 与 INT8** 两种精度下的端到端时延与微架构计数器（INT8 是下一代 FP8/FP4 在现有硅上唯一可实测的密度近亲）。核心结论：

> **ET-BERT 在 910B 上既非算力受限、也非 HBM 带宽受限，而是受限于 ① host/kernel 调度开销 与 ② 片上 MTE2（把权重/激活填入 L1）。** 因此单纯堆"更高算力的 FP4"与"更大 HBM 带宽"几乎无效——实测把精度从 FP16 降到 INT8，端到端反而**慢 26%**。只有先压低调度开销、加快片上填充，FP4 才能兑现约 **20%** 的收益。

---

## 一、实验平台、方法与口径

**被测芯片（910B4）结构要点**：24 个 AI Core，每个 AI Core 含一个 **CUBE 矩阵单元（16×16×16 MAC）** 与 2 个向量单元（AIV）。存储层级 **L0A/B/C → L1（512KB/核）→ L2（96MB 共享）→ HBM（392 GB/s）**。峰值 FP16 280 TFLOPS、INT8 560 TOPS。AI Core 内部按 **pipe** 计时：`mac`（CUBE 计算）、`mte1`（L1↔L0 搬运）、`mte2`（HBM/L2→L1 填充）、`fixpipe`（L0C→输出/requant）；AIV 侧有 `vec/mte2/mte3`。

**精度类比**：FP8/FP4 无对应硅，用 910B 的 FP16/INT8 做**数据密度**类比（INT8≙FP8 1 字节、INT4≙FP4 0.5 字节），算力倍数则按下一代假设外推。

**测量手段（都是厂商原生工具，与本项目模型无关，可独立复现）**：
- `ais_bench`：端到端 NPU 计算时延（batch 1–64）。
- `msprof` 的 aic-metrics：`ArithmeticUtilization`（CUBE 利用率 cube_util_fp16/int8）、`PipeUtilization`（各 pipe 耗时）、`L2Cache`（per-op 命中率）、`Memory` + `--sys-hardware-mem`（L1/L2/HBM 带宽、HBM 与 LLC 命中率）。
- `AMCT` PTQ：把 ET-BERT 量化为 INT8 后再 ATC 转 OM 实测（填补"INT8 从未实测"的空白）。

> 关于"MFU/CUBE 利用率"：本文统一指 **CUBE 实际做有用 MAC 的时间占比**，即 msprof 的 `aic_mac_*_ratio` 按算子耗时加权。

## 二、贯穿全文的关键发现：瓶颈定位

实测 ET-BERT FP16 端到端时延随 batch 变化：

| batch | 1 | 4 | 16 | 64 |
|---|---|---|---|---|
| 总时延 (ms) | 1.75 | 2.35 | 4.96 | 14.86 |
| 每样本 (ms) | **1.75** | 0.59 | 0.31 | 0.23 |

线性拟合：**边际计算 0.21 ms/样本，固定开销 1.55 ms**。即 b1 时 **88% 时延是与 batch 无关的固定开销**，仅 12% 是真正的算。这 1.55ms 主要来自 **~330 个 kernel 的 host 派发 + 大量小 kernel 的 fill/drain 延迟**。

两个独立的真机计数器共同锁定瓶颈：

1. **CUBE 利用率低且 AIC 内主导 pipe 恒为 mte2**——见下表，说明设备侧卡在"把数据填进 L1"，不是卡在算。
2. **HBM 带宽利用率仅 0.4%**——中央内存几乎闲置，带宽根本不是瓶颈。

| 真机指标 (FP16) | b1 | b16 | 含义 |
|---|---|---|---|
| CUBE 利用率 | 32% | 75% | 远未跑满 |
| AIC 主导 pipe | mte2 | mte2 | 卡在 L1 填充 |
| L2 读命中率 | 74% | 96% | 复用被片上接住 |
| HBM 带宽利用率 | 0.4% | 0.4% | HBM 几乎不动 |

> **结论：瓶颈是 host 调度 + 片上 MTE2(L1 填充)，不是算力、也不是 HBM 带宽。** 下面四问的答案都由此推出。

## 三、四个问题的回答

### Q1：混合精度下 MFU 怎么公平计算？

**结论**：混合精度没有统一峰值，**必须按各算子各自精度的峰值分别归一，再按耗时加权**，即 `MFU = Σ(t_op/T)·cube_util_op`（= msprof 的 `cube_util_fp16 + cube_util_int8`）；跨精度别直接比 MFU 数值（分母峰值变了），要比就比**有效吞吐/时延**。

- **实测**：CUBE 利用率 FP16 32%→75%（b1→b16）、INT8 14%→46%。
- **原因**：INT8 峰值更高（560 vs 280 TOPS），同样 MAC 量更早算完、CUBE 更早空闲，**利用率天生更低**——所以跨精度直接比利用率会得出"低精度更差"的误判。

### Q2：800T FP4 与 200T FP8 的访存带宽需求一样吗？

**结论：不一样，且要区分"字节量"与"带宽速率"。**

| 口径 | FP8→FP4 | 原因 |
|---|---|---|
| 搬运字节量 | **减半** | 位宽减半 |
| 维持峰值所需带宽 (GB/s) | **翻倍** | 算力 4× 仅被密度 2× 抵消一半 |

- **量化**：维持 compute-bound 的最小带宽——FP8 74 → FP4 **147 GB/s**（正好 2×）。对照 910B FP16→INT8（算力 2× / 密度 2×，正好抵消）→ 带宽需求**不变**。
- **判据**：`BW需求 ∝ 峰值算力 × 位宽`；算力倍数与密度倍数是否相等，决定带宽需求是否变化。FP4 算力(4×)跑赢密度(2×)，故翻倍。

### Q3：FP8→FP4，MFU 会下降、需更大带宽吗？

**结论：会下降、确需更大带宽——而且真机直接证伪了"降精度必提速"：FP16→INT8 端到端反而慢 26%。**

- **实测**：同 batch INT8 CUBE 利用率 < FP16（b16 46% vs 75%）；INT8 的 `mac` 时间确为 FP16 的一半（2× 吞吐属实）；但端到端 **INT8 2.21ms > FP16 1.76ms**。
- **原因**：ET-BERT 非算力受限，砍掉的 mac 时间被 host/MTE2 占据的 wall-clock 吸收；更关键的是**量化部署引入 requant 算子（AscendQuant + 融合 dequant），实测占 INT8 wall 的 19–35%**（落在 fixpipe/AIV 侧），等于给本已碎片化的调度再加一批小 kernel。
- **机理澄清**：MFU 下降的主因是 **roofline 左移（compute 时间缩得比访存快→CUBE 被饿）**，而非"tile 量化损失"——ET-BERT 归约维全为 64 的整数倍，tile 量化效率 ≈ 1，无损。

### Q4：16MB Cache 兜得住模型吗？加 Cache 降带宽？真机命中率/DDR 利用率？精度变化？对齐？

**结论：兜不住整模型但无需兜；真机显示 96MB L2 已吸收绝大部分复用，HBM 近乎闲置。**

- **容量**：OM（含权重）FP16 **165MB** / INT8 **85MB**，16MB 装不下整模型；但推理逐层流式，真正需常驻的是**单层工作集**（FP16 14MB / INT8 7MB），仓库够放一层即可。
- **首次实测的命中率与带宽利用率**（本项目此前为空白）：

| 精度 | L2 读命中率 (per-op) | HBM 带宽利用率 | 三级带宽 |
|---|---|---|---|
| FP16 | 74–96% | **0.4%** | HBM 1.4 → L2 2.7 → L1 122 GB/s |
| INT8 | **95–99.8%** | 0.4% | （复用全被 L1/L2 接住） |

- **加 Cache 能降带宽吗**：仅在"装下单层工作集"之前有效；装下一层后，单流推理不会回用上层权重，再加几乎无增益。
- **精度变化的影响**：位宽减半→单层工作集更小→**更易常驻 L2→命中率更高**（INT8 明显高于 FP16）→ HBM 更闲。**低精度让带宽更省，而非更紧。**
- **对齐**：910B 走 fractal/NZ 格式、**512B 对齐**；低精度通过加深 fractal 内维（FP16 16×16 / INT8 16×32 / FP4 16×64）保持 512B。只要归约维是 32/64 整数倍（ET-BERT 全满足），**FP4 下零对齐浪费**。
> 口径修正：AIC 的 L2 读命中（74–99%）才是解释 HBM 闲置的指标；系统级 `llc_read_write` 计数器另读约 46%，为更窄的 LLC 结构，二者口径不同。

## 四、下一代架构探索：低精度到底值不值？

把以上真机结论外推到两项下一代改动——**①专用电路吸收量化/反量化开销（理想 fixpipe）；②HBM 带宽翻倍**。投影以实测为锚：

| 场景 | 时延 (ms) | 相对 FP16 |
|---|---|---|
| FP16（实测） | 1.76 | 1.00× |
| INT8（实测，含 requant） | 2.21 | **1.26×（更慢）** |
| INT8 + 理想 fixpipe | 1.43 | 0.82× |
| **FP4 + 理想 fixpipe** | **1.40** | **0.80×（快 20%）** |
| **FP4 + 理想 fixpipe + 2× 带宽** | 1.40 | **2× 带宽贡献 ≈ 0** |

**两条核心结论**：
1. **专用低精度电路（消除 requant 开销）能把 FP4 从负收益救回 +20%**，值得做；
2. **4× 算力与 2× 带宽几乎全浪费**——因瓶颈在 host 调度与片上 MTE2，不在算力或 HBM 带宽。

瓶颈链一览：

```
瓶颈链：    host 调度  ≫  片上 MTE2(L1 填充)  >  量化 requant  ≫  HBM 带宽(仅 0.4%)
谁吃掉收益： ↑未解决        ↑未解决               ↑下一代电路可去   ↑根本不是瓶颈
能兑现：     FP4 理想 ≈ 比 FP16 快 20%；4× 算力 / 2× 带宽 ≈ 0 收益
```

## 五、下一代架构建议（按投入产出排序）

### 必做（攻 88% 固定开销，收益最大）
1. **整图下沉 / 静态图执行**：消除逐 kernel 的 host 派发。
   *依据*：大 kernel 模型 host_gap 仅 8.4µs/核，而 ET-BERT 这类小 kernel 高达 41.7µs/核——调度在小 kernel 上被完全暴露。
2. **算子融合**：FlashAttention（QK^T+softmax+AV）、融合 QKV、融合 FFN+激活、融合 LayerNorm+残差，把 kernel 数从 330 降到约 110。
   *量化（架构模型）*：kernel→1/3 时 wall 约 ×0.38；叠加整图下沉 → **约 ×0.13（≈7× 提速）**。
3. **攒批 / continuous batching**：摊薄固定开销。
   *依据*：b1 固定开销占 88%，b64 降到 10%，吞吐 571→4306 样本/秒。

### 有效（攻片上 MTE2，第一步做完才显效）
4. **加快 L2→L1 填充带宽 + 增加 MTE2 引擎/DMA 队列深度**——这是真正卡住的搬运路。
   *注意*：实测 L1↔L0（mte1）仅用约 6% 峰值带宽，**加 L1↔L0 带宽无效**；要加的是"填 L1"的 L2→L1 路。架构模型证实：l1_l0_bw 翻倍 → AIC 时间不变；mte2 路带宽翻倍 → AIC 时间减半。
5. **增大 L1 + 改进 tiling（weight-stationary，提高每次填充的复用行数）**，减少 MTE2 重复填充。

### 有限 / 可缓做
6. **专用低精度电路（消除 requant）**：能救回 +20%，但排在 1–5 之后。
7. **堆 FP4 4× 算力 / 2× HBM 带宽**：本模型上**几乎零收益**，除非前面已拆掉调度与片上搬运两道墙。

> **给架构师**：先拆 host 调度墙与片上 MTE2 墙，再谈低精度与带宽；顺序反了，FP4 的 4× 算力只是纸面富贵。

## 六、一页纸总结

| 问题 | 结论 | 关键实测证据 |
|---|---|---|
| Q1 公平算 MFU | 各精度按各自峰值归一、按耗时加权；勿跨精度直接比 | CUBE 利用率 FP16 32→75%、INT8 14→46% |
| Q2 FP4 带宽需求 | 字节量减半，但所需带宽翻倍 | 维持满载 FP8 74 → FP4 147 GB/s |
| Q3 降精度代价 | MFU 降、需更大带宽；实测 INT8 反慢 26% | 1.76→2.21ms；requant 占 19–35% |
| Q4 Cache 与带宽 | 装不下整模型但无需；HBM 仅 0.4%，低精度命中更高 | L2 命中 FP16 74–96% / INT8 95–99.8% |
| 下一代 | 收益被 host+MTE2 吃掉，先拆墙再降精度 | FP4 理想 1.40ms（快 20%），2× 带宽 ≈ 0 |

**最终结论**：910B 上 ET-BERT 不缺算力、不缺 HBM 带宽，缺的是"更少的 kernel 调度"和"更快的 L1 填充"。把这两件事做好，下一代 FP4 才有意义。

### 附：实验完整性说明
FP16/INT8 全部为 910B4 真机实测（ais_bench + msprof + AMCT）。FP8/FP4 为外推（无硅）。"理想 fixpipe"通过真机 per-op OP-Type 分解剥离 requant 算子得到，非纯建模。"整图下沉/融合"的提速倍数为架构模型投影 + 实测 kernel 数与 host_gap 关系推算；端到端 SDPA 融合真机实测因平台环境（transformers 与 ONNX 导出器不兼容）暂未完成，列为后续补测项。
